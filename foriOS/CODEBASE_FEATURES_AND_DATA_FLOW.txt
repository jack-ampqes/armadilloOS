2. AUTHENTICATION & AUTHORIZATION
================================================================================

--- 2.1 AUTH FLOW ---

- Login: POST /api/auth/login
  - Body: { email, password }
  - Passwords are hashed with SHA-256 (crypto.createHash('sha256').update(password).digest('hex')).
  - User is looked up in Supabase table "users" (id, email, password_hash, name, role).
  - On success: response sets two cookies and returns { token, user: { id, email, name, role } }.
  - Cookies set:
    - auth_token: HTTP-only, 7-day maxAge, path /
    - user_info: NOT httpOnly (so client can read role), JSON string of { id, email, name, role }
  - Login page also writes token, user_email, user_name, user_role to localStorage (redundant with user_info cookie).

- Logout: POST /api/auth/logout
  - Clears auth_token and user_info cookies (maxAge: 0).
  - Client should clear localStorage and redirect to /login.

- Signup: POST /api/auth/signup
  - Body: { email, password, role? } (role: Admin | Sales Rep | Distributor | Technician; default Distributor).
  - Validates email format, password length >= 6, checks for existing user in "users".
  - Inserts into Supabase "users" (email, password_hash, role). Uses supabaseAdmin first if SUPABASE_SERVICE_ROLE_KEY is set, else supabase.
  - Same cookie/session setup as login; user is logged in after signup.

--- 2.2 ROLES & PERMISSIONS ---

- Defined in src/lib/permissions.ts.
- Roles: Admin | Sales Rep | Distributor | Technician
- Permissions: FullAccess, OrdersViewing, InventoryViewing, InventoryEditing, QrCodesBarcodes, Quoting, QuotingCustomItems, ManufacturerOrders
- Matrix (summary):
  - Admin: all permissions (FullAccess = true).
  - Sales Rep: InventoryViewing, Quoting only.
  - Distributor: same as Sales Rep.
  - Technician: OrdersViewing, InventoryViewing, InventoryEditing, QrCodesBarcodes, Quoting.
- normalizeRole() maps legacy strings (e.g. "rep", "sales_rep") to canonical roles; unknown → Distributor.
- Client: usePermissions() in src/lib/usePermissions.ts reads role from user_info cookie via getRoleFromUserInfoCookie() and exposes role + hasPermission(permission).

--- 2.3 API AUTH HELPERS (src/lib/auth.ts) ---

- requireAuthWithRole(req): Reads auth_token and user_info cookies; returns { user } or { response: 401/403 }. Used where any logged-in user is enough.
- requirePermission(req, permission): Same as above but then checks hasPermission(user.role, permission); 403 if insufficient.
- requireAdmin(req): Requires role === 'Admin'; 403 otherwise. Used for admin-only routes.

Route protection is done per API route (no single middleware.ts found); each protected route calls one of these helpers and returns the error response if auth fails.

--- 2.4 NAVIGATION & DASHBOARD VISIBILITY ---

- ConditionalNavigation hides nav on /login and /signup; otherwise shows Navigation.
- Navigation (src/components/Navigation.tsx): ALLOWED_NAV_BY_ROLE limits which nav items are shown:
  - Admin: all (Dashboard, Orders, Quotes, Inventory, Codes, Customers, Sales Reps, Distributors, Alerts, Documents, Users, Company Profiles).
  - Sales Rep / Distributor / Technician: only /, /inventory, /alerts.
- Dashboard (page.tsx): DASHBOARD_SECTIONS_BY_ROLE filters section cards the same way (Admin sees all sections; others only Inventory and Alerts).
- ConditionalMain: main content has lg:ml-[280px] and padding except on /login and /signup (no padding there).

5. API ROUTES – COMPLETE LIST WITH METHOD, AUTH, AND DATA FLOW

--- 5.1 AUTH ---

- POST /api/auth/login
  - No auth. Body: { email, password }. Reads users from Supabase; verifies password_hash (SHA-256); sets auth_token + user_info cookies; returns { token, user }.

- POST /api/auth/logout
  - No auth. Clears auth_token and user_info cookies.

- POST /api/auth/signup
  - No auth. Body: { email, password, role? }. Inserts users in Supabase; sets same cookies as login; returns { token, user }.

--- 5.2 DASHBOARD ---

- GET /api/dashboard/stats?period=
  - No explicit requireAuth in code shown; typically called when user is logged in. period: thisMonth | last3Months | last6Months | ytd | lastYear | allTime.
  - Data flow: getDefaultShopifyCredentials() (Supabase shopify_connections or env); getOrders() from Shopify for date range; aggregates revenue, pending orders, revenue trend, top products, status distribution; reads armadillo_inventory.inventory for low/out of stock counts; reads quotes for active (DRAFT/SENT) count; top customers from order data. If QuickBooks connected (getDefaultQuickBooksCredentials), fetches P&L for this month and this year via getProfitAndLoss/parseProfitAndLoss. Returns combined JSON. If Shopify is not configured, returns partial data (inventory, quotes, QuickBooks only) with error message.

--- 5.3 ORDERS (SHOPIFY) ---

- GET /api/orders
  - requirePermission(OrdersViewing). Query: limit, status, fulfillment_status, financial_status, created_at_min, created_at_max.
  - Data flow: getDefaultShopifyCredentials() → getOrders() from Shopify API; transforms to app format (id: shopify-{id}, orderNumber, status, totalAmount, customer, orderItems, shopifyData). Returns array.

- POST /api/orders
  - requirePermission(OrdersViewing). Body: orderItems, customerEmail, customerPhone, shippingAddress, billingAddress, notes, tags, salesRepName, completeOrder.
  - Data flow: getDefaultShopifyCredentials() → createDraftOrder() with line items (variantId or custom title/price/sku); optionally completeDraftOrder() if completeOrder true. Returns created draft or completed order.

- GET /api/orders/by-number/[number]
  - Fetches single order by order number (from Shopify).

- GET /api/shopify/orders (if present): likely raw Shopify orders list.
- GET /api/shopify/orders/[id]: single Shopify order.

--- 5.4 INVENTORY ---

- GET /api/inventory?sku=
  - requirePermission(InventoryViewing). Reads armadillo_inventory.inventory; optional filter by sku. Transforms with parseSkuToProduct (sku-parser) for product name/category; returns { inventory: [...] }.

- POST /api/inventory
  - requirePermission(InventoryEditing). Body: { sku, quantity } (quantity is delta). Updates inventory row (quantity += delta), writes to inventory_history (source: 'manual', user_id, user_email). Then checkLowStockAlerts(). Returns updated inventory object.

- GET /api/inventory/history?sku=&limit=
  - requirePermission(InventoryViewing). Reads armadillo_inventory.inventory_history; optional sku filter; limit max 500. Returns { history: [...] }.

--- 5.5 PRODUCTS ---

Else: supabase.rpc('get_products') or get_product_by_sku(sku); transforms to { id, name, sku, price, ... }; returns array or { products } for single sku.

- POST /api/products
  - Body: sku, price, color, leadtime, quantity. Inserts into armadillo_inventory.inventory (sku, price, color, leadtime, quantity). Returns created row.

- GET /api/products/[id]: single product (detail).

--- 5.6 CODES (QR / BARCODE) ---

- GET /api/codes?sku=&format=&type=
  - No requireAuth in snippet. sku optional. format: dataURL | svg; type: qr | barcode.
  - If sku: loads one row from armadillo_inventory.inventory; generates QR (URL to /inventory/scan?sku=) or barcode (CODE128) via qrcode/jsbarcode/canvas; returns { sku, productName, code, qrCode, barcode?, format, type }.

--- 5.7 QUOTES ---

- GET /api/quotes
  - requirePermission(Quoting). supabaseAdmin.from('quotes').select('*, quote_items(*)').order('created_at', { ascending: false }). Maps with mapQuoteRowToApi (quote-supabase). Returns array.

- POST /api/quotes

Generates quote_number via generateQuoteNumber() (YY + NNNN); inserts quotes + quote_items; runs checkQuoteExpirationAlerts(); then getDefaultQuickBooksCredentials() and pushQuoteToQuickBooks(); updates quote with quickbooks_estimate_id and quickbooks_synced_at. Returns created quote (or with warning if QB push fails).

- GET /api/quotes/[id]

9. DATA FLOW SUMMARY (WHERE DATA LIVES AND GOES)
================================================================================

- Orders: Source of truth = Shopify. App reads via Shopify API (getOrders, getOrder); creates via draft orders. No local orders table required for Shopify flow; dashboard stats and reports/sales use Shopify order data.
- Inventory: Source of truth = Supabase schema armadillo_inventory (inventory table). Quantity updates write to inventory_history. Low/out-of-stock alerts written to public.alerts. Product names can come from SKU parser or DB.
- Quotes: Source of truth = Supabase (quotes + quote_items). Optional sync to QuickBooks (Estimates); quickbooks_estimate_id and quickbooks_synced_at stored on quote. Quote numbers generated from DB (YY + NNNN).
- Customers, Distributors, Sales reps: Supabase public schema (customers, distributors, sales_reps). Distributor addresses geocoded to lat/long (Nominatim) and stored on distributor; map API returns locations for pins.
- Manufacturers & manufacturer orders: armadillo_inventory (manufacturers, manufacturer_orders, manufacturer_order_items). Tracking uses FedEx API when carrier is FedEx and credentials set.
- Users & auth: Supabase users table; session in cookies (auth_token, user_info). Companies in companies table; users can have company_id for icon/logo.
- Documents: Supabase storage bucket "documents" + documents table (metadata). Admin only.
- Alerts: Supabase alerts table; created/updated by checkLowStockAlerts and checkQuoteExpirationAlerts (and optionally pending orders).
- Reports: Sales = derived from Shopify orders. Inventory = derived from armadillo_inventory.inventory. Quotes = derived from quotes/quote_items. No separate report storage; computed on demand.
- Products: Fetched via RPC (get_products, get_product_by_sku) or from Shopify when source=shopify. New products can be inserted into armadillo_inventory.inventory via POST /api/products.

11. EXACT RESPONSE SHAPES (FOR CODABLE MODELS / PARSERS)
================================================================================

--- Naming convention ---
- API responses that go through explicit transforms use camelCase in JSON (inventory, quotes, dashboard stats, alerts, etc.). You can infer Swift Codable / Kotlin data classes directly from the samples below.
- Supabase tables are snake_case. Endpoints that return raw rows (e.g. some admin lists) may expose snake_case unless the route maps to camelCase (e.g. quote-supabase mapQuoteRowToApi). When in doubt, check the route: if it calls mapQuoteRowToApi, keysToCamel, or a similar transform, the response is camelCase.

12. DESIGN TOKENS (DARK THEME)
================================================================================

--- Colors ---
- Background (page): #181818 (used on body in layout; overrides CSS variable for main surface).
- Background (CSS variable): --background: hsl(0, 0%, 9%) ≈ #171717 (card/popover/muted/input use same or similar).
- Card / secondary surface: same as background (0 0% 9%); cards often use border border-white/20 and sometimes bg-[#181818] or gradient (e.g. from-[#1f1f1f] to-[#252525]).
- Text primary: white (#ffffff) — --foreground: 0 0% 100%.
- Text secondary / muted: white at 60% opacity (text-white/60, text-white/50, text-white/40). In CSS vars: --muted-foreground: 0 0% 60% ≈ #999999.
- Border: --border: 0 0% 20% ≈ #333333; in Tailwind, border-white/20 is common.
- Accent (primary actions): white background, dark text (e.g. bg-white text-[#181818]). No single bright accent hex; primary is inverted (white on dark).
- Destructive (errors, delete, “Out of Stock”): --destructive: 0 84% 60% (red). Badge destructive: #C23B22, hover #AC3834. Inline error text: text-red-400, text-red-300; error cards: border-red-500/50, bg-red-500/10.
- Success (“In Stock”, positive P&L): Badge success #3FA34D, hover #7FB446. Text: text-green-400.
- Warning (“Low Stock”, “Pending”, “Expiring”): Badge warning #F0C42D, hover #FBAF00. Text: text-yellow-400.
- Status semantics in UI:
  - In Stock / Resolved / positive net income: green (green-400 or success badge).
  - Low Stock / Pending / Expiring soon: yellow (yellow-400 or warning badge).
  - Out of Stock / Critical / negative net income / errors: red (red-400 or destructive badge).

--- Typography ---
- Font stack: Body uses Montserrat (weights 300, 400, 500, 600, 700) — var(--font-montserrat). Headings (h1–h6) use Orbit — var(--font-orbit), font-size 1.5rem. Both loaded from Google Fonts in layout.tsx.
- Additional families available as CSS variables (used where needed): --font-geist-sans (Geist), --font-geist-mono (Geist Mono), --font-michroma (Michroma), --font-orbit (Orbit).
- Body: font-weight 300, font-style normal. No explicit body font-size in globals; default (16px) applies. Titles in cards/sections often use text-xl or text-2xl (1.25rem / 1.5rem), font-semibold. Caption/secondary: text-sm, text-white/60 or text-white/50.
- Summary: Title/heading = Orbit 1.5rem (and larger with Tailwind classes). Body = Montserrat 300. Caption = same family, text-sm, muted color. No “use system (SF)”; the design system uses Google Fonts (Orbit + Montserrat) as above.

--- Radii / spacing ---
- --radius: 0.75rem (12px); used for card and component corners. Badge: rounded-lg (8px).
- Sidebar width: 280px (w-[280px]). Main content: lg:ml-[280px], padding p-8, top padding pt-24 (mobile) / lg:pt-8 (desktop).
